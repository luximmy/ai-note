# Insight Note 编辑器核心技术踩坑与重构实录

## 1. 类型系统与业务建模重构

在项目初期，我们将所有文本类的区块统一定义为 `TextBlock`。随着业务逻辑复杂化，这种模糊的建模导致了类型安全隐患。

- **踩坑点：模糊状态导致的脏数据**
  - **现象**：TypeScript 允许给段落（paragraph）设置标题层级（level），导致渲染逻辑出现不可预知的分支。
  - **解决方案**：实施“消除不可能状态 (Make Impossible States Impossible)”原则。将 `TextBlock` 拆分为独立的 `ParagraphBlock` 和 `HeadingBlock` 接口。
  - **收益**：实现了精确的类型收窄，UI 层代码提示更准确，彻底杜绝了非法属性组合。

## 2. 渲染架构：从 Switch-Case 到表驱动

随着区块类型增加，核心渲染器 `BlockRenderer` 变得异常臃肿。

- **踩坑点：违反开闭原则 (OCP)**
  - **现象**：每次新增区块类型（如 Todo, Image）都必须修改核心调度文件中的 `switch` 分支，极其容易引发代码冲突。
  - **解决方案**：引入“区块注册表 (Block Registry)”模式。使用对象映射字典将 `type` 字符串直接映射到对应的 React 组件。
  - **收益**：将组件分发的复杂度从 O(N) 降至 O(1)，新增区块仅需在注册表中添加一行，无需触动调度逻辑。

## 3. 富文本引擎（Tiptap）深度适配 — 技术预研

> **注意**：以下内容为 Tiptap 接入前的技术预研结论，记录了在调研阶段识别出的关键坑位。Tiptap 尚未实际集成到项目中，当前区块编辑器为轻量级结构化输入方案。

将第三方重量级库集成进 Next.js App Router 遇到了多次运行环境冲突。

- **关键坑位 A：SSR 水合不匹配 (Hydration Mismatch)**
  - **报错**：`Tiptap Error: SSR has been detected...`
  - **成因**：Next.js 在服务端预渲染 HTML，而 Tiptap 试图在无 DOM 环境下计算排版，导致两端生成的 HTML 不一致。
  - **解决方案**：在 `useEditor` 配置中显式设置 `immediatelyRender: false`，强制其在客户端挂载后再启动。

- **性能坑位 B：编辑器频繁销毁重载**
  - **现象**：打字时输入法中断、光标突然丢失。
  - **成因**：React 重新渲染导致传给 Tiptap 的回调函数引用变化，触发引擎销毁重刷。
  - **解决方案**：应用“最新引用模式 (Latest Ref Pattern)”。使用 `useRef` 缓存 `onUpdate` 回调，结合 `useEffect` 动态更新 `current` 指向，确保 Tiptap 拿到的函数地址永远固定。

## 4. 全栈数据流与容灾逻辑

这是编辑器交互体验最核心的部分，涉及并发渲染与网络同步的深度权衡。

- **核心抉择：`useOptimistic` 的能力边界**
  - **结论**：这不是 `useOptimistic` 的缺陷，而是“事务型乐观更新”与“长会话编辑输入”在语义上的错配。
  - **冲突**：`useOptimistic` 更适合“单次、离散、可快速收敛”的原子操作；在“高频连续打字 + 防抖保存”中，Transition 与防抖时序容易错位，导致 UI 短暂回跳（闪烁）。
  - **最终方案**：采用“双缓冲状态 (Double Buffering)”作为编辑器主策略。手动维护 `blocks`（活跃 UI 状态）与 `safeSnapshot`（服务端确认状态），配合 `use-debounce` 实现抗压同步与可控回滚。
  - **策略补充**：`useOptimistic` 仍用于点赞、勾选、单次提交等离散操作，不在本项目的富文本连续输入链路中强制使用。

- **容灾闭环：15% 异常率下的优雅回滚**
  为了保证用户数据的最终一致性，我们构建了以下链路：
  1.  **用户输入**：`useState` 立即更新，0 延迟反馈。
  2.  **网络同步**：`Debounced Server Action` 后台静默保存，防止 DDOS 服务器。
  3.  **后端报错**：`catch` 捕获异常 ➔ `Toast` 实时警告 ➔ `setBlocks(fallback)` 状态回滚 ➔ `router.refresh()` 强制数据对齐。
  4.  **状态劫持**：在 `RichTextEditor` 内部监听 `initialContent` 变化，发现回滚时显式调用 `editor.commands.setContent()` 强刷编辑器内容。

## 5. 高并发输入与状态合并陷阱 (Pending Merge)

在引入 `Partial<Block>` 支持多字段更新（如 Todo 块的文本与勾选状态并存）后，暴露出基于区块编辑器的致命并发问题。

- **核心坑位 A：跨区块编辑互相污染（全局防抖的灾难）**
  - **现象**：用户在 800ms 内快速修改 Block A 后立刻修改 Block B，Block A 的数据会丢失，且只有 Block B 被提交。
  - **成因**：调度中心只有一个全局的 `debouncedServerUpdate` 定时器。连续操作会不断重置定时器，导致前置操作被无情抛弃；且成功后会将全量 `latestBlocks` 设为安全快照，导致未真正提交的脏数据被误认为“安全”。
  - **解决方案**：引入“块级防抖 (Block-level Debouncing)”。使用 `useRef<Record<string, Timeout>>` 为每一个 `blockId` 分配独立的防抖定时器，确保区块生命周期完全隔离。快照推进时，严格通过 `blockId` 匹配只更新对应的局部状态。

- **核心坑位 B：同区块闭包状态覆盖 (脏写问题)**
  - **现象**：在 Todo 块中，输入一段文字后立刻点击勾选，网络请求最终只发送了 `checked` 状态，刚输入的文字神秘消失。
  - **成因**：由于异步闭包与 800ms 防抖窗口的存在，最后一次操作（点击勾选）产生的 Payload 覆盖了前一次（文本输入）的 Payload。
  - **解决方案**：引入“挂起更新合并缓冲区 (Pending Merge Buffer)”。建立 `pendingUpdatesRef`，在 800ms 窗口期内的所有 `updates` 不会直接发起请求，而是被深度合并（Deep Merge）并暂存。定时器结束时，一次性提取合并后的“终态数据”发送至服务端，彻底阻断高频微操作导致的属性丢失。

- **衍生坑位 C：幽灵定时器与内存泄漏**
  - **现象**：快速输入后立刻点击侧边栏切换笔记，控制台抛出 React `unmounted component` 警告，甚至把旧笔记的数据保存到了新路由下。
  - **成因**：组件卸载时，字典中挂载的 `setTimeout` 并未被销毁。
  - **解决方案**：在 `useEffect` 的 cleanup 函数中，通过 `Object.values(timersRef.current).forEach(clearTimeout)` 执行严格的内存回收。

## 6. TypeScript 高阶类型博弈

随着 `BlockRegistry`（区块注册表）的完善，我们遭遇了 TypeScript 类型系统在动态分发场景下的理论边界。

- **坑位：动态分发中的 `never` 悖论**
  - **报错**：`Type 'any' is not assignable to type 'never'`。
  - **成因**：在 `.map` 遍历 `blocks` 渲染时，TypeScript 无法将联合类型（`block.type`）与上下文动态绑定（相关联类型丢失）。为了绝对安全，编译器强制要求传入的 props 必须是所有区块组件接口的交叉类型（Intersection Type）。由于各区块的 `type` 字面量互斥，交叉结果直接坍缩为 `never`。
  - **解决方案**：采用“定义处严控，调用处豁免”的实用主义策略。
    1. 在 `BlockRegistry` 定义时，利用泛型与 `Extract` 锁住映射关系，确保特定 `type` 只能绑定特定组件。
    2. 在动态调用 `<TargetComponent />` 处，使用 `as React.FC<any>` 进行有限断言，把类型豁免范围压缩在单一调用点，避免污染整条类型链路。

## 7. 总结

通过本次重构，Insight Note 从“能编辑”提升到“在高并发输入下仍具备一致性保障”。当前方案在实践上形成了三层防线：块级隔离调度、窗口期更新合并、异常场景回滚与对齐。与此同时，类型系统采用“定义强约束 + 调用点局部豁免”的策略，在工程可维护性与开发效率之间取得了可落地的平衡。

## 8. 乱序响应回归坑（阶段二收口）

在完成块级防抖与合并缓冲后，仍出现“旧请求后返回并覆盖新状态”的经典并发问题。

- **现象**：同一区块连续两次编辑，第二次请求先返回；稍后第一条旧请求返回并把 UI/快照写回旧值。
- **根因**：防抖只解决“发出多少请求”，不保证“响应按顺序回来”。
- **解决方案**：在保存链路引入全局递增请求序号，并按 `blockId` 维护最近成功序号。若当前响应序号落后于已成功序号，直接丢弃，不推进安全快照。
- **当前状态**：该策略已在 `BlockRenderer` 中落地，成为连续编辑场景的必备防线。

此前“可运行但不可验证”，导致回归问题只能靠手测捕捉，效率低且不稳定。

- **补齐内容**：已落地 `loading`、`error`、保存成功、保存失败回滚四条最小回归路径测试。
- **工程收益**：后续重构保存链路和区块注册表时，可通过自动化测试快速发现行为回归。
